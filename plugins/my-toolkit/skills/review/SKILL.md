# Review - 审查

结构化的审查工作流，支持多种审查类型。代码、Spec 和计划审查采用 3 agent 并发审查模式。

## 触发方式

用户执行 `/my-toolkit:review` 命令时激活。传入审查目标作为参数。

## 工作流程

### 1. 确定审查类型

根据用户输入判断审查类型：

- **代码审查** — 审查指定文件或目录的代码质量
- **Spec 审查** — 审查需求规格文档的完整性和一致性
- **计划审查** — 审查实现计划的可执行性
- **文档审查** — 审查文档的准确性、完整性和规范性
- **PR 审查** — 通过 gh 命令获取 PR 变更进行审查
- **设计审查** — 审查前端 UI/UX 设计质量

如果范围不明确，主动询问用户。

### 2. 插件与技能准备

根据审查涉及的技术栈，检查并加载所需插件和技能：

- **通用**：`context7-plugin` — 查阅技术栈最新文档和 API，贯穿审查全周期
- **前端项目必选**：`frontend-design` — Web 页面设计；`ui-ux-pro-max` — 企业级 UI/UX 设计标准（需要时）
- **React 技术栈**：按需加载 `vercel-react-best-practices`（性能优化）、`vercel-composition-patterns`（组件组合模式）、`vercel-react-native-skills`（React Native 开发）
- **UI 组件库**：按需加载 `antd`（Ant Design 组件使用）、`ant-design`（Ant Design 架构决策与主题定制）、`shadcn`（shadcn/ui 组件管理）

### 3. 并发执行审查（并发3次）

#### 选择审查技能

**代码审查 / PR 审查：**
- 启用 `requesting-code-review`技能
- 获取 base 和 head 的 git SHA
- 前端项目审查时加载 `web-design-guidelines` — 基于 Web Interface Guidelines 进行 UI/UX 合规审查。

**Spec 审查：**
- 启用 `requesting-spec-review`技能

**计划审查：**
- 启用 `requesting-plan-review`技能

**文档审查 / 设计审查（直接分析）：**

- **文档审查**：读取文档内容，对照代码验证准确性
- **设计审查**：读取前端代码和组件结构

#### 审查结果处理

根据审查类型，使用对应的并发审查模式，使用 Agent 工具同时派发 3 个审查 agent，每个 agent 仅输出审查发现的问题，不做修复。
所有 agent 全部审查完成后，按照以下原则合并审查结果：

- **共识检测**：多个 agent 同时发现的问题 → 高置信度优先处理；仅 1 个 agent 发现的 → 验证后再处理
- **去重**：相同根因的不同表述合并为一个；同一文件/函数的重叠问题合并为单个修复
- **修复顺序**：共识问题优先 → Critical → Important → Minor，每类问题修复后运行验证

#### 审查关注点及处理方式

不同审查类型的关注点及处理方式：

**代码审查维度：**
- 安全漏洞（注入、XSS、敏感信息等）和功能缺陷（潜在 bug、边界条件） → 立即修复
- 性能瓶颈或资源泄漏（错误处理缺失等） → 评估影响后修复
- 代码异味、违反 SOLID/DRY/KISS 原则、可读性差 → 重构修复
- 代码风格和规范不一致 → 对齐项目规范
- 超出需求范围的变更 → 跳过（回归到需求边界内）
- 审查 agent 误解上下文的误报 → 跳过并附理由

**Spec 审查维度：**
- 完整性缺失（TODOs、占位符、TBD、验收标准缺失） → 立即补充
- 章节间矛盾或逻辑不一致 → 对齐统一后修复
- 需求模糊不清、可能导致错误实现 → 要求澄清或补充细节
- 范围蔓延、包含未请求的功能（YAGNI） → 移除多余内容
- 审查 agent 对业务背景理解偏差的误判 → 跳过并附理由

**计划审查维度：**
- 占位符、TBD、不完整步骤 → 立即补充具体内容
- Spec 需求无对应任务 → 补充遗漏任务
- 任务边界模糊、步骤不可执行 → 拆分或重写任务
- 零上下文工程师无法执行 → 补充必要的上下文说明
- "后续实现"、"补充细节"等模糊描述 → 替换为明确行动项
- 审查 agent 对技术方案理解的误判 → 跳过并附理由

**文档审查维度：**
- 内容与代码行为不一致 → 立即修正
- 结构缺失或逻辑混乱 → 重组文档结构
- 示例代码无法运行 → 修正或补充可运行的示例
- 格式不规范、排版不一致 → 统一格式
- 遗漏或过时信息 → 更新至最新状态
- 审查 agent 对文档上下文理解的误判 → 跳过并附理由

**PR 审查维度：**
- 破坏性变更或安全漏洞 → 立即修复
- 变更范围不合理（过大或与 PR 描述不匹配） → 标记并建议拆分或对齐
- 测试覆盖不足 → 补充必要测试
- 与 PR 描述不匹配的低优先级变更 → 评估后处理或跳过
- 审查 agent 误解 PR 上下文的误报 → 跳过并附理由

**设计审查维度：**
- 无障碍合规性缺失 → 立即修复
- 交互不合理或用户体验问题 → 评估影响后修复
- 视觉层次混乱或信息架构不清 → 重构优化
- 响应式适配缺失 → 补充适配方案
- 设计系统不一致 → 对齐设计规范
- 审查 agent 对设计意图误解的误判 → 跳过并附理由

### 4. 审查问题修复

根据审查类型，启用对应的 `receiving-*-review` 技能处理反馈。

- 验证反馈是否与代码库实际情况一致
- 对不明确或有疑问的反馈先澄清，再实施
- 逐条处理 Critical 和 Important 问题，每条处理后运行验证
- 如有无法自动解决的矛盾，呈现给用户决策

**代码审查 / PR 审查：**

- 启用 `receiving-code-review`技能

**Spec 审查：**

- 启用 `receiving-spec-review`技能

**计划审查：**

- 启用 `receiving-plan-review`技能

**文档审查 / 设计审查（直接输出）：**

按严重程度分类输出审查报告：

```
## 审查报告

### 严重问题（必须修复）
- [问题说明及修复建议]

### 建议优化（推荐改进）
- [优化建议及理由]

### 亮点（做得好的地方）
- [值得肯定的做法]
```

### 5. 确认和后续

将审查结果呈现给用户，询问是否需要：
- 针对某个问题展开详细说明
- 直接修复严重问题
- 生成优化代码

## 注意事项

- 审查意见要具体可执行，避免空泛的"建议优化"
- 既要指出问题，也要肯定好的做法
- 严重问题标明原因和风险，给出具体修复方案
- 不提出超出审查范围的修改建议
- 遵循项目现有的代码规范和风格
