# PRD - 需求文档

引导式创建产品需求文档，从需求分析到文档输出。

## 触发方式

用户执行 `/my-toolkit:prd` 命令时激活。传入产品或功能名称作为参数。

## 工作流程

### 1. 需求探索

> **superpowers 技能**：启用 `brainstorming`，通过系统化提问澄清需求意图。

通过交互式提问收集产品需求：

- 探索项目上下文，理解现有产品定位
- 逐个提出澄清问题：产品/功能名称、目标用户、使用场景
- 提出 2-3 个产品方案并分析权衡
- 逐节呈现需求设计，每节获得用户认可后再继续
- 核心功能和优先级
- 技术约束和非功能性需求
- 竞品参考或灵感来源

### 2. 需求分析

整理和分析收集到的信息：

- 提炼核心用户故事
- 识别功能模块和依赖关系
- 确定 MVP 范围和迭代计划
- 评估技术可行性

### 3. 文档编写

按照标准 PRD 结构输出文档：

```markdown
# 产品需求文档 (PRD)

## 1. 概述
- 产品名称
- 产品定位和目标
- 目标用户

## 2. 用户故事
- 核心用户场景
- 用户旅程

## 3. 功能需求
- 功能列表和优先级（P0/P1/P2）
- 功能详细描述
- 业务规则和约束

## 4. 非功能性需求
- 性能要求
- 安全要求
- 兼容性要求

## 5. 数据模型
- 核心数据实体
- 实体关系

## 6. 接口设计（概要）
- 核心接口列表
- 关键交互流程

## 7. 里程碑和排期
- MVP 范围
- 迭代计划
```

### 4. 确认和调整

将文档呈现给用户，根据反馈调整完善。

## 注意事项

- 需求描述要具体可执行，避免模糊表述
- 优先级划分要合理，MVP 聚焦核心价值
- 非功能性需求要根据实际情况设定，不过度要求
- 保持文档简洁，避免大段空洞描述
